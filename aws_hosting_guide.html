HTML → AWS S3 — 무엇이 생성되고 왜 URL이 만들어지는가

deploy.js가 AWS에 실제로 한 일과 URL 생성 원리
A. 전체 구조
로컬 PC ai_clean_check.html
deploy.js
→
AWS S3 버킷 파일 저장소
+ 웹서버 역할
→
AWS DNS 버킷명으로
URL 자동 부여
→
브라우저 전 세계 누구나
URL로 접속
B. deploy.js가 AWS에서 실제로 한 일 (5단계)
1
S3 버킷 생성 — AWS에 이름이 있는 저장 공간을 만듦
CreateBucketCommand → 버킷명: ai-clean-check-2026 / 리전: 서울(ap-northeast-2)
결과: AWS 서버 어딘가에 "ai-clean-check-2026"라는 저장공간이 생성됨
2
퍼블릭 액세스 차단 해제 — 기본적으로 S3는 외부 접근을 막아둠
PutPublicAccessBlockCommand → BlockPublicPolicy: false
결과: 인터넷에서 이 버킷에 접근 가능한 상태로 전환
3
정적 웹 호스팅 활성화 — S3를 단순 저장소가 아닌 웹서버처럼 동작하도록 설정
PutBucketWebsiteCommand → IndexDocument: ai_clean_check.html
결과: AWS가 이 버킷에 전용 웹 주소(URL)를 자동으로 부여
→ 이 단계에서 URL이 생성됨
4
버킷 정책 적용 — "누구나 파일을 읽을 수 있다"는 권한 규칙 등록
PutBucketPolicyCommand → s3:GetObject / Principal: "*"
결과: 인터넷의 모든 사람이 파일 읽기(다운로드·열기) 가능. 쓰기·삭제는 불가
5
HTML 파일 업로드 — 로컬 파일을 버킷 안에 저장
PutObjectCommand → ai_clean_check.html → S3 버킷 내부
결과: URL로 접속하면 이 파일이 브라우저에 전달됨
C. URL(주소)이 만들어지는 원리
http://ai-clean-check-2026.s3-website.ap-northeast-2.amazonaws.com
버킷명 ai-clean-check-2026 — 내가 지정한 이름. 전 세계에서 유일해야 함
서비스 s3-website — "이 버킷은 정적 웹 호스팅 중"을 나타내는 AWS 고정 키워드
리전 ap-northeast-2 — 서울 리전. 실제 파일이 저장된 AWS 데이터센터 위치
도메인 amazonaws.com — AWS 소유 도메인
💡 IP가 아닌 이유: 이 URL은 고정 IP가 아니라 DNS 이름입니다. 브라우저가 이 주소를 입력하면 AWS의 DNS 서버가 실제 서버 IP로 변환해 연결합니다. AWS 내부 IP는 수시로 바뀔 수 있지만 URL은 항상 같게 유지됩니다. 버킷명을 바꾸면 URL도 바뀌고, 버킷명이 같으면 언제나 같은 URL이 생성됩니다.
D. 켜기 / 끄기
▶ 켜기 · 배포 · 수정 반영
node deploy.js
B의 1~5단계를 자동 실행.
이미 버킷이 있으면 생성을 건너뛰고
설정 복원 + 파일 업로드만 수행.
⏹ 끄기 (파일은 S3에 보존)
node site-off.js
버킷 정책 삭제 + 웹 호스팅 비활성화.
URL 접속 시 403 오류 표시.
파일은 삭제되지 않으므로 켜기 즉시 복구.
E. 다른 HTML 파일을 배포할 때

deploy.js 상단의 FILE 값만 바꾸고 실행하면 됩니다.

const FILE = './다른파일.html';   // ← 이 줄만 변경
